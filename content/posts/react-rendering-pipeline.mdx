## 为什么你需要一张渲染故障地图

大多数性能问题并不是单点 bug，而是多个环节叠加：

- 数据拉取抖动
- 组件重渲染级联
- Hydration mismatch
- 样式计算与布局回流

先把故障拆到阶段，再优化代码，效率会高很多。

## 三层定位法

### 1. 先看用户体感

通过 Web Vitals 先确认问题落在哪条链路：

```ts
export function markHydration(name: string) {
  performance.mark(`${name}:hydration:start`);
  queueMicrotask(() => {
    performance.mark(`${name}:hydration:end`);
    performance.measure(
      `${name}:hydration:cost`,
      `${name}:hydration:start`,
      `${name}:hydration:end`
    );
  });
}
```

### 2. 再看组件热区

把耗时组件按交互路径归类，而不是按目录归类。这样可以直接回答一个问题：
哪个页面路径在真实使用中最痛。

### 3. 最后看构建产物

Chunk 拆分不合理常常会导致首屏慢和长任务。把关键路径脚本预算写进 CI。

## 常见误区

> 不要把“减少 re-render”当成终极目标。真正目标是降低用户端感知延迟。

当你强行做 memo 化，却忽略了数据流和网络层，通常只是把复杂度藏起来。

## 一个可执行检查清单

1. 关键路径组件是否可被服务器提前渲染。
2. 交互热区是否存在不必要的 state 提升。
3. 样式变更是否触发连续布局回流。
4. 产物体积是否超出预算阈值。

当这四项可以自动化检测后，性能问题会从“事后救火”变为“发布前阻断”。
